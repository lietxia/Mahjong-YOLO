# Mahjong-YOLO 升级到 YOLO26 计划书

## 1. 文档目的

本计划书用于指导当前 **Mahjong-YOLO** 项目从现有的 **YOLOv11 / Ultralytics** 训练与验证体系，升级到当前官方主线中的 **Ultralytics YOLO26**。

这份计划书只聚焦以下问题：

- YOLO26 到底应如何理解；
- 当前项目升级到 YOLO26 的正确路径是什么；
- 哪些资产可以复用，哪些部分必须重训或重做；
- 升级时应如何处理数据集、类别配置、训练验证、导出部署和回滚。

---

## 2. 关键结论

### 2.1 “YOLO 26”在这里具体指什么

结合当前官方文档、官方发布和主线代码，本文所说的“YOLO 26”应理解为：

> **Ultralytics YOLO26 模型族**

而不是：

- `ultralytics` Python 包版本 26；
- 一个与 YOLOv11 输出完全兼容、只需替换权重的小版本；
- 一个可以直接复用旧检测头和旧后处理逻辑的无缝升级。

### 2.2 对当前项目最重要的判断

对这个麻将牌检测项目而言，最安全的升级方式不是“直接升版复用旧权重”，而是：

> **先冻结数据与类别定义，再基于 YOLO26 预训练权重做受控微调，并以当前 YOLOv11 基线作为对照组。**

### 2.3 本项目是否适合升级到 YOLO26

适合，但前提是先解决当前仓库里已经存在的几个基础问题：

- 类别 schema 不一致；
- 数据配置文件不一致；
- 训练、验证、展示脚本对类别表的依赖并不统一；
- 若后续依赖 ONNX / 浏览器 / 其他部署链路，则必须重新验证输出契约。

---

## 3. 项目背景

### 3.1 当前项目现状

当前 Mahjong-YOLO 仓库本质上是一个：

> **以 Python + Ultralytics + Notebook 为中心的模型训练、验证、导出与演示仓库**

目前可确认的现状包括：

- 训练与实验主要通过 Notebook 和 Python 脚本完成；
- 现有主模型为 `trained_models_v2/` 下的 YOLOv11 系列权重；
- 已存在 `.pt`、`.onnx`、`.mlpackage` 等导出资产；
- 当前评估、演示、验证脚本都围绕 YOLOv11 风格的 Ultralytics 工作流构建。

### 3.2 数据集来源

本项目当前使用的数据来源为：

- **Kaggle Dataset**：<https://www.kaggle.com/datasets/shinz114514/mahjong-hand-photos-taken-with-mobile-camera/data>

从项目定位看，这是一套面向**手机拍摄麻将手牌照片**的目标检测数据集，天然具有以下特点：

- 目标尺寸偏小；
- 牌面之间高度相似；
- 存在遮挡、叠放、光照变化、视角变化；
- 错标和框偏移会被模型明显放大。

因此，这类任务对**类别定义一致性**和**标注质量**的敏感度，通常高于通用物体检测任务。

---

## 4. 升级目标

本次升级的目标，不是单纯“把模型名字从 YOLOv11 改成 YOLO26”，而是完成以下四件事：

1. 在统一的数据与类别配置前提下，建立 YOLO26 的可训练基线；
2. 判断 YOLO26 在麻将牌小目标场景中是否真实优于当前 YOLOv11 基线；
3. 验证 YOLO26 在现有导出与部署链路中是否可用；
4. 在满足精度、稳定性和导出兼容性要求后，决定是否替换默认模型。

---

## 5. 升级范围

### 5.1 本次计划包含

- 数据与类别 schema 统一；
- YOLO26 训练实验；
- 基线对照验证；
- 导出兼容性验证；
- 模型切换与回滚策略设计。

### 5.2 本次计划不包含

- 重做整个数据集；
- 重新设计标注体系以外的业务逻辑；
- 大规模产品前端改造；
- 脱离 Ultralytics 工作流自研训练框架；
- 直接承诺“YOLO26 一定比 YOLOv11 更好”。

---

## 6. 升级前提与核心风险

## 6.1 最大前提：先统一类别 schema

当前项目里已经能观察到多套不一致的类别定义，这会直接破坏升级结论的可信度。

升级前必须先明确：

- 最终类别总数到底是多少；
- 是否保留 `UNKNOWN`；
- 各类别的顺序和 ID 映射是否固定；
- 训练、验证、展示、导出是否全部使用同一份类别字典。

### 升级前必须产出

- `classes.json` 或等价唯一类别映射文件；
- 一个唯一可信的 `data.yaml`；
- 一套固定 train / val / test 划分说明。

## 6.2 数据质量风险

Kaggle 数据集虽然可用，但对麻将牌任务来说，仍需重点检查：

- 标注框是否过松或过紧；
- 是否存在遮挡牌漏标；
- 是否存在近重复图片跨训练集与验证集；
- 是否存在相似类别错标；
- 是否存在类别分布严重不均衡。

## 6.3 输出与部署契约风险

YOLO26 的默认推理与导出范式相较 YOLOv11 风格已经发生变化，尤其是在 end-to-end 检测输出上。

这意味着：

- 旧的后处理逻辑不能默认继续使用；
- ONNX / TensorRT / 浏览器端 / 自定义 parser 都要重新验证；
- 导出成功不代表现有业务链路完全兼容。

---

## 7. 升级策略总原则

本项目的升级应遵循以下原则：

### 原则 1：先冻结基线，再开始升级

先把当前最可信的 YOLOv11 模型、验证脚本、导出结果和评估指标固定下来，作为不可变对照组。

### 原则 2：先解决 schema，再比较模型

如果类别定义和数据配置不统一，那么任何“YOLO26 提升了多少”的结论都不可信。

### 原则 3：先跑通标准训练，再谈深度调参

第一轮只做最少变量变更：

- 同一数据集；
- 同一类别表；
- 同一划分；
- 同级别模型尺寸；
- 接近当前评估协议。

### 原则 4：先验证训练效果，再验证部署链路

先确认 YOLO26 在离线验证中有价值，再投入时间做导出、推理适配和部署重构。

### 原则 5：升级失败时必须可回退

升级不是一次性替换，而应是一个随时可切回 YOLOv11 基线的受控实验过程。

---

## 8. 可复用资产与必须重做项

## 8.1 可复用资产

以下内容可以继续复用：

- 现有数据目录组织方式；
- 现有 train / val / test 划分思路；
- 现有验证脚本思路；
- 现有指标汇总格式；
- 现有模型导出流程框架；
- 现有 ONNX / 部署验证习惯；
- 现有多模型尺寸对照方式。

## 8.2 必须重做或至少重微调

以下内容不应假设能直接沿用：

- 现有 YOLOv11 最终检测权重；
- 现有检测头权重；
- 旧版输出 shape 假设；
- 旧版后处理逻辑；
- 旧版阈值配置一定仍然最优；
- 旧版导出与部署适配代码一定仍可直接使用。

---

## 9. 分阶段执行计划

## Phase 0：升级准备与基线冻结

### 目标

让升级实验具备可信对照组。

### 任务

1. 确定当前默认 YOLOv11 基线模型。
2. 重新跑一遍现有验证，记录基线指标。
3. 固化类别字典和唯一 `data.yaml`。
4. 固化 train / val / test 划分。
5. 固化当前导出方式与推理阈值。

### 产物

- 基线权重记录
- 基线验证报告
- `classes.json`
- 统一后的 `data.yaml`
- 基线导出产物清单

### 验收标准

- 当前 YOLOv11 结果可重复复现；
- 类别定义在训练、验证、导出、展示中完全一致。

---

## Phase 1：数据集体检与修正

### 目标

在开始 YOLO26 训练前，先排除最容易干扰结论的数据问题。

### 任务

1. 检查是否存在跨划分近重复图片。
2. 检查是否存在错标、漏标、边框偏移。
3. 检查类别分布是否严重失衡。
4. 抽样检查遮挡场景与密集手牌场景。
5. 确认是否保留 `UNKNOWN` 类。

### 产物

- 数据集质量检查报告
- 纠错建议清单
- 最终类别与划分版本说明

### 验收标准

- 能明确说明数据是否可直接用于 YOLO26 实验；
- 已知高风险样本被记录，而不是隐藏在训练噪声里。

---

## Phase 2：建立 YOLO26 最小训练基线

### 目标

在不大改流程的前提下，让 YOLO26 在当前任务上先跑通。

### 任务

1. 安装支持 YOLO26 的 Ultralytics 版本。
2. 选择与当前基线同级别的 YOLO26 模型尺寸。
3. 使用官方预训练权重初始化。
4. 采用统一后的麻将数据集做第一次 fine-tune。
5. 保持与 YOLOv11 基线尽量接近的训练与评估条件。

### 模型尺寸建议

优先做：

- 一个小模型；
- 一个中模型。

不建议第一轮就全尺寸铺开，更不建议第一轮直接上最大模型。

### 产物

- YOLO26 初版训练权重
- 第一轮验证指标
- 与 YOLOv11 的同条件对比结果

### 验收标准

- YOLO26 训练可稳定跑通；
- 验证流程可正常输出；
- 结果可与 YOLOv11 同口径比较。

---

## Phase 3：小目标与高相似度场景专项调优

### 目标

针对麻将牌任务本身的特点，评估 YOLO26 是否真正适合这个场景。

### 任务

1. 检查密集牌面场景的召回率。
2. 分析高相似度类别的混淆情况。
3. 分析遮挡场景的误检与漏检。
4. 评估输入尺寸对小目标的影响。
5. 调整增强策略，避免对麻将牌规则矩形目标造成破坏。

### 重点指标

- 总体 mAP
- Recall
- Precision
- 易混类别对误分类率
- 密集手牌场景漏检率
- 低清晰度样本表现

### 产物

- 小目标专项验证报告
- 混淆分析报告
- 调优建议清单

### 验收标准

- 升级收益不仅体现在总体指标，也能体现在麻将任务关键场景上；
- 不是单纯“总分略高，但业务场景更差”。

---

## Phase 4：导出与部署兼容性验证

### 目标

确认 YOLO26 不只是“训练有效”，而且“现有部署链路也可接入”。

### 任务

1. 导出 YOLO26 模型到 ONNX。
2. 检查输出 shape 与坐标语义是否变化。
3. 检查是否仍需旧版 NMS / decode 流程。
4. 检查现有脚本、浏览器方案或其他运行时是否需要适配。
5. 做同图推理对照，比较 YOLOv11 与 YOLO26 的输出差异。

### 产物

- 导出兼容性报告
- 输出契约说明文档
- 需要适配的部署清单

### 验收标准

- 导出产物可用；
- 输出张量含义清楚；
- 后处理是否需要重写有明确结论。

---

## Phase 5：决策与切换

### 目标

在有完整证据的前提下，决定是否把默认主模型切换到 YOLO26。

### 切换条件

只有同时满足以下条件，才建议切换默认模型：

1. YOLO26 在统一 test 集上整体不低于当前 YOLOv11 基线；
2. 麻将牌关键场景表现不差于基线；
3. 导出与部署链路可用；
4. 推理时延、模型大小和部署成本在可接受范围内；
5. 回滚方案已经准备好。

### 产物

- 升级决策报告
- 默认模型切换说明
- 回滚方案文档

### 验收标准

- 是否升级有明确、可复查的依据；
- 切换不是“靠感觉”，而是基于训练、验证和部署三层证据共同决定。

---

## 10. 训练策略建议

### 推荐路径

建议采用以下训练策略：

1. 使用 **YOLO26 官方预训练权重** 初始化；
2. 用统一后的麻将数据集做 **全量 fine-tune**；
3. 第一轮尽量少改变量；
4. 确认有效后再做专项调优。

### 不建议路径

以下做法不建议作为第一选择：

- 直接拿 YOLOv11 最终权重强迁到 YOLO26；
- 在 schema 未统一时大规模开实验；
- 同时大改数据、模型、超参、导出方式；
- 训练还没证明有效就开始做大规模部署适配。

---

## 11. 模型尺寸选择建议

对于这个麻将牌检测项目，建议按“小 + 中”双路线推进：

### 小模型

适合：

- 验证升级可行性；
- 控制训练与部署成本；
- 接近未来轻量部署场景。

### 中模型

适合：

- 作为主要精度候选；
- 平衡小目标能力与部署成本。

### 大模型

只建议在以下情况下再考虑：

- 小模型和中模型都明显不足；
- 遮挡与密集场景依然表现差；
- 部署成本不是当前主约束。

---

## 12. 验收标准

升级是否成功，不能只看单个 mAP 数字，建议按四层验收：

### 12.1 训练层

- 模型训练稳定；
- 指标可重复；
- 不依赖单次幸运结果。

### 12.2 业务层

- 易混类别误判率可接受；
- 密集手牌场景召回率达标；
- 遮挡与低清晰度场景不明显退化。

### 12.3 导出层

- ONNX 导出成功；
- 输出含义明确；
- 后处理适配完成。

### 12.4 成本层

- 模型大小可接受；
- 推理延迟可接受；
- 部署链路复杂度没有失控。

---

## 13. 回滚策略

升级必须带回滚。

建议保留以下资产作为冻结版本：

- 当前 YOLOv11 默认权重；
- 当前 `data.yaml` 与类别映射快照；
- 当前验证报告；
- 当前导出结果；
- 当前推理阈值配置。

### 回滚定义

回滚不应是“重新训练旧模型”，而应是：

> **直接切回旧配置、旧权重、旧导出产物与旧验证基线。**

---

## 14. 最终建议

对这个 Mahjong-YOLO 项目来说，升级到 YOLO26 的正确顺序应当是：

1. 统一类别 schema；
2. 冻结 YOLOv11 基线；
3. 做数据质量检查；
4. 用 YOLO26 预训练权重建立同口径训练基线；
5. 做麻将牌小目标专项分析；
6. 验证导出与部署兼容性；
7. 满足验收条件后再切换默认模型。

一句话总结：

> **这次升级本质上不是“升个版本号”，而是一次“受控的模型族替换实验”。**

如果前置数据和类别问题不先解决，那么 YOLO26 的潜在收益会被无效实验噪声掩盖；如果这些前提先收敛，YOLO26 才有机会在麻将牌这种小目标、高相似度场景里体现真实价值。
