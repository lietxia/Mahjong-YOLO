# Mahjong-YOLO 改造为 WASM + WebGPU 项目的详细计划

## 1. 计划目标

本计划的目标不是继续扩展 Python 脚本，而是把当前以 **Python + Ultralytics YOLOv11** 为中心的麻将牌识别仓库，逐步改造成一个面向浏览器运行的 **WASM + WebGPU 推理项目**。

这里的“改造”采用最现实、最稳妥的解释：

> **保留 Python 侧训练与模型导出能力，把推理执行、预处理、后处理和可视化迁移到 Web 端。**

也就是说，短中期内不会把训练本身搬到浏览器；重点是把“**部署运行时**”从 Python 切换到浏览器。

---

## 2. 为什么采用这个方向

从当前仓库现状出发，这条路线最合理，原因有三点：

1. 当前仓库已经具备 `.onnx` 模型资产，天然适合做浏览器推理改造。
2. 当前 Python 代码主要承担的是训练、验证、导出、展示，并没有成熟的服务端应用层；因此直接走浏览器端更符合“轻部署演示 + 跨平台访问”的目标。
3. WebGPU 负责高性能推理，WASM 负责补足浏览器端的高效 CPU 逻辑，两者组合比“纯 JS + 纯 Canvas”更接近工程可用方案。

---

## 3. 改造后的目标形态

## 3.1 最终目标

将项目重构为两个相互配合的子系统：

### A. 模型工厂层（保留 Python）

职责：

- 数据处理
- 模型训练
- 模型验证
- 模型导出（重点是 ONNX）
- 类别、阈值、输入尺寸等元数据产出

### B. Web 推理层（新增浏览器工程）

职责：

- 在浏览器中加载模型
- 调用 WebGPU 执行推理
- 用 WASM 或高性能 TS 实现预处理 / 后处理
- 对图片或视频帧进行检测
- 在 Canvas/WebGL/WebGPU 叠加检测结果
- 向用户展示类别、置信度、框选结果与性能指标

---

## 4. 推荐的目标技术架构

## 4.1 推荐栈

对于这个项目，建议优先采用如下技术组合：

- **前端框架**：Vite + TypeScript
- **推理运行时**：ONNX Runtime Web（优先使用 **WebGPU EP**，必要时保留 WASM fallback）
- **图像预处理**：TypeScript 实现为主，性能瓶颈部分再下沉到 WASM
- **后处理**：TypeScript/WASM 实现 YOLO 输出解码、阈值过滤、NMS、类别映射
- **渲染层**：Canvas 2D 起步，后续可按性能需要切到 OffscreenCanvas
- **线程模型**：主线程负责 UI，Web Worker 负责推理调度
- **模型与配置资源**：静态资源托管 + 版本化元数据 JSON

## 4.2 为什么不建议第一阶段直接自己手写 WebGPU 推理内核

虽然目标名义上是 “WASM + WebGPU 项目”，但第一阶段不建议直接从零写一套自定义 WebGPU 张量计算框架，原因是：

- 成本极高。
- 调试困难。
- 与现有 ONNX 导出资产衔接差。
- 风险远高于收益。

因此第一阶段最现实的方案是：

> **模型执行交给成熟的浏览器推理运行时，WASM 主要承担性能敏感的 CPU 逻辑，WebGPU 主要承担神经网络推理。**

这仍然是一个标准意义上的 **WASM + WebGPU** 项目，只是避免了不必要的底层重复造轮子。

## 6. 必须先解决的前置问题

在开始 Web 工程之前，必须先把当前仓库里几个基础问题统一掉。

## 6.1 统一类别定义

当前仓库中存在 37 类、38 类、`UNKNOWN`、`class_0` 等不一致情况。改造前必须明确：

- 最终类别总数到底是多少。
- 类别顺序是否固定。
- 是否保留 `UNKNOWN`。
- 红五（`0m / 0p / 0s`）在训练和推理中是否单独作为类别。

### 交付物

- 一份唯一可信的 `classes.json` 或 `labels.json`
- 一份统一后的训练配置
- 一份前端可直接使用的类别映射文件

### 验收标准

- Python 验证脚本、导出脚本、Web 前端读取到的类别顺序完全一致。

## 6.2 固化模型输入输出约定

需要固定以下内容：

- 输入尺寸，例如 `640 x 640`
- 输入通道顺序，例如 RGB / BGR
- 像素归一化规则
- 输出张量形状
- 类别概率与框坐标的解析方式
- NMS 参数

### 交付物

- `model_manifest.json`
- 文档化的输入输出协议说明

### 验收标准

- 浏览器端在同一张图上的解码结果与 Python 端趋势一致。

## 6.3 明确主模型选择

当前仓库存在 `n / s / m` 三个主模型。改造第一阶段不应全部一起做，而应选一个作为主线。

### 建议

- **Phase 1 首选 `yolo11n_best`**：体积最小，更适合浏览器首屏加载、内存占用与兼容性验证。
- **`yolo11s_best` 作为精度对照与第二阶段候选**：当 Web 端闭环稳定后，再评估是否升级为默认模型。
- `yolo11m_best` 暂不作为首发默认模型，避免首屏加载和显存压力过大。

这样做的核心原则是：

> **首阶段优先验证浏览器交付链路，而不是一上来就追求最佳综合精度。**

## 7. 目标系统的运行链路

改造完成后的浏览器推理链路建议如下：

```text
用户选择图片 / 摄像头帧
    ↓
浏览器读取图像
    ↓
预处理（resize / normalize / tensorize）
    ↓
WebGPU 执行 ONNX 模型推理
    ↓
后处理（decode / score filter / NMS / class map）
    ↓
Canvas 叠加框、类别名、置信度
    ↓
输出检测结果与性能指标
```

其中职责拆分建议如下：

- **WebGPU**：模型推理主计算
- **WASM**：可选承担高成本预处理、后处理、NMS 或图像转换
- **TypeScript**：资源调度、UI、状态管理、结果渲染、配置管理

## 8. 分阶段实施计划

## Phase 0：仓库基线整理

### 目标

让当前 Python 仓库达到“可稳定导出 Web 资产”的状态。

### 任务

1. 统一类别定义。
2. 清理并确认最终使用的 `data.yaml`。
3. 确认首发模型（建议 `yolo11n_best`）。
4. 重新导出一份用于 Web 的标准 ONNX。
5. 固化模型元数据：输入尺寸、类别、阈值、导出版本。
6. 明确现有 `models/**/*.onnx` 仅作为参考，不直接视为正式发布资产。

### 交付物

- 标准化 ONNX 模型
- `classes.json`
- `model_manifest.json`
- 基线图片集与 Python 参考输出

### 验收标准

- 基线图片在 Python 侧可稳定复现。
- 导出的 ONNX 在离线 ONNX Runtime 环境中可运行。

---

## Phase 1：建立最小可运行 Web 推理 Demo

### 目标

做出一个最小可运行的浏览器版本，完成“上传图片并检测麻将牌”。

### 任务

1. 创建前端工程（Vite + TypeScript）。
2. 接入浏览器端推理运行时。
3. 加载首发 ONNX 模型。
4. 实现图片上传。
5. 实现基础预处理。
6. 实现模型输出解析。
7. 实现基础 NMS。
8. 用 Canvas 绘制框和标签。

### 范围边界

第一阶段**不做**：

- 摄像头实时流
- 多模型切换
- 自定义 WebGPU kernel
- 复杂 UI 设计
- 离线缓存优化

### 交付物

- 一个能在浏览器打开的 Demo 页面
- 图片上传检测流程
- 展示类别、置信度和检测框

### 验收标准

- 在基线样例图上能输出检测结果。
- 浏览器输出与 Python 基线在主要目标上基本一致。
- 模型能在支持 WebGPU 的浏览器中跑通。

---

## Phase 2：把预处理 / 后处理工程化

### 目标

让推理结果从“能跑”提升到“结构清晰、结果稳定、方便维护”。

### 任务

1. 把预处理逻辑从页面代码中抽离为独立模块。
2. 把 YOLO 输出解码抽离为独立模块。
3. 把 NMS 做成独立实现，并补充单元测试。
4. 建立统一的类别映射与颜色映射。
5. 对齐 Python 参考结果，做误差比对。

### 可选增强

如果纯 TypeScript 后处理性能不足，再引入 WASM：

- 将 NMS 下沉到 WASM
- 将图像格式转换下沉到 WASM
- 将高频数组运算下沉到 WASM

### 交付物

- `preprocess` 模块
- `postprocess` 模块
- `labels` / `manifest` 模块
- 基线比对测试

### 验收标准

- 模块职责清晰。
- 主要基线图输出可重复。
- 后处理逻辑不再散落在 UI 层。

---

## Phase 3：引入 Worker 与性能优化

### 目标

解决主线程阻塞问题，让体验更接近真实应用。

### 任务

1. 将模型加载和推理迁移到 Web Worker。
2. 使用消息机制传递图像与检测结果。
3. 将渲染与推理解耦。
4. 增加推理耗时、预处理耗时、后处理耗时监控。
5. 评估是否需要 OffscreenCanvas。

### 交付物

- Worker 化推理通路
- 性能面板或调试指标输出

### 验收标准

- 推理期间页面交互不明显卡顿。
- 能看到分阶段耗时数据。

---

## Phase 4：加入实时视频 / 摄像头能力

### 目标

从静态图检测升级到实时场景。

### 任务

1. 接入摄像头流。
2. 控制帧采样频率。
3. 限制并发推理，避免堆帧。
4. 增加暂停 / 恢复 / 截图能力。
5. 针对低端设备做降级策略。

### 交付物

- 实时检测页面
- 设备性能分级策略

### 验收标准

- 摄像头场景可稳定运行。
- 页面不会因连续推理而失去响应。

---

## Phase 5：产品化与交付优化

### 目标

让项目从技术 Demo 走向可发布、可维护。

### 任务

2. 静态资源缓存策略。

4. WebGPU 不可用时的降级方案。
5. 错误提示、加载态、空状态完善。
6. 文档与部署说明整理。

### 交付物

- 可部署版本

- 降级策略说明

### 验收标准

## 9. 代码结构改造建议

建议最终将仓库整理为类似结构：

```text
.

│   ├── notebooks/
│   ├── scripts/
│   ├── trained_models/
│   └── manifests/
├── web_app/                    # 新增浏览器推理工程
│   ├── src/
│   │   ├── components/
│   │   ├── workers/
│   │   ├── inference/
│   │   │   ├── preprocess/
│   │   │   ├── runtime/
│   │   │   ├── postprocess/
│   │   │   └── labels/
│   │   └── assets/
```

这里最重要的思想是：

- **训练与部署分层**
- **模型资产与前端代码分层**
- **预处理 / 推理 / 后处理分层**

## 10. WASM 在这个项目里的具体角色

为了避免“WASM 只是口号”，需要明确它在本项目中的职责边界。

建议的 WASM 角色如下：

### 10.1 第一优先级：性能敏感后处理

适合放到 WASM 的内容：

- NMS
- 大量框候选的筛选与排序
- 密集数组扫描

### 10.2 第二优先级：图像预处理热点逻辑

适合放到 WASM 的内容：

- resize 前后的像素搬运
- 格式转换
- 批量归一化

### 10.3 不建议第一阶段放进 WASM 的内容

- 整个 UI 层
- 所有业务状态管理
- 复杂的模型加载编排

原因很简单：这些部分不属于真正的性能热点，而且会显著增加开发与调试复杂度。

---

## 11. WebGPU 在这个项目里的具体角色

WebGPU 的职责应保持单纯：

> **优先承担神经网络推理计算。**

第一阶段建议通过成熟运行时间接使用 WebGPU，而不是直接手写所有计算图。这样做有几个好处：

- 与 ONNX 导出路径更匹配。
- 更快得到可验证结果。
- 更利于和 Python 基线做一致性比对。

---

## 12. 风险清单

## 12.1 类别定义不一致风险

这是当前仓库最现实、最直接的风险。若不先统一，前端结果即使推理正确，也可能显示错误标签。

## 12.2 ONNX 导出兼容性风险

并不是所有导出的 ONNX 都能直接在浏览器侧高效执行，需要验证：

- 算子兼容性
- 输出形状是否稳定
- 是否需要额外简化或重导出

## 12.3 浏览器显存与加载时间风险

`yolo11m_best` 体积较大，若直接作为默认模型，可能导致：

- 首次加载慢
- WebGPU 初始化压力大
- 低端设备体验差

## 12.4 预处理 / 后处理不一致风险

如果浏览器端图像处理规则与 Python 端不一致，会导致检测质量明显下降。

## 12.5 WebGPU 可用性风险

不同浏览器、系统、驱动对 WebGPU 支持不完全一致，因此必须准备降级路径。

---

## 13. 建议的验收策略

整个改造不应只看“页面能跑”，而应按以下层次验收。

### 13.1 模型层验收

- ONNX 模型可加载
- 输出张量形状稳定
- 关键样例图结果接近 Python 基线

### 13.2 推理层验收

- WebGPU 路径能跑通
- fallback 路径可说明
- 单张图推理耗时可观测

### 13.3 展示层验收

- 检测框位置正确
- 类别名映射正确
- 置信度显示正确

### 13.4 工程层验收

- 前端工程可独立启动
- 模型与元数据可独立发布
- 文档清楚说明使用方式和限制

---

## 14. 推荐的实施顺序

如果只考虑“最快做出正确版本”，推荐顺序如下：

1. 统一类别与配置。
2. 固化首发模型与 ONNX 导出流程。
3. 建立最小 Web Demo。
4. 对齐 Python 与浏览器结果。
5. 抽离后处理模块。
6. 引入 Worker。
7. 再决定哪些热点真正值得下沉到 WASM。
8. 最后再做实时视频与产品化优化。

这条路径的核心原则是：

> **先保证结果正确，再追求性能；先跑通链路，再下沉到底层。**

---

## 15. 第一阶段明确不做的事情

为了让计划可执行，必须明确第一阶段不做什么：

- 不把训练搬到浏览器。
- 不重写 YOLO 训练框架。
- 不手写整套 WebGPU 深度学习内核。
- 不同时支持全部模型规模。
- 不在首版就覆盖所有浏览器兼容问题。
- 不先做复杂 UI 再补推理。
- 不承诺与 Python / Ultralytics 指标逐项完全一致，只承诺在基线样例上达到可接受误差范围内的一致性。

---

## 16. 最终建议

对于这个仓库，最合适的改造方向不是“把 Python 项目硬改成前端项目”，而是：

1. **把 Python 仓库保留为模型工厂。**
2. **新增一个真正的 Web 推理工程。**
3. **以 ONNX 为桥梁，把推理运行时迁移到 WebGPU。**
4. **只把真正的性能热点放进 WASM。**
5. **先用 `yolo11n_best` 做浏览器首发验证，再以 `yolo11s_best` 作为质量增强路线。**

如果按这个计划推进，那么最终得到的会不是一个“名义上的 wasm+webgpu 项目”，而是一个结构清楚、可逐步上线、并且与现有 YOLOv11 资产衔接顺畅的浏览器麻将识别项目。
