# Mahjong-YOLO 项目技术说明

## 1. 项目概述

本项目是一个基于 **YOLOv11** 的麻将牌识别仓库，当前核心目标是：

- 对真实拍摄场景中的麻将牌进行目标检测与类别识别。
- 维护不同规模的 YOLOv11 模型权重，覆盖速度与精度的不同取舍。
- 提供训练、验证、推理示例，以及导出到 **ONNX** / **CoreML** 的脚本。

从当前仓库实况看，它更接近一个“**以 Python + Notebook 为中心的模型实验与导出仓库**”，而不是一个已经产品化的前后端一体化应用。

---

## 3. 技术栈总览

### 3.1 编程语言与运行环境

- **Python**：当前仓库的主语言。
- **Jupyter Notebook**：用于数据处理、标注辅助、训练、预测与实验记录。
- 当前仓库中**没有看到** `requirements.txt`、`pyproject.toml`、`Pipfile`、`package.json` 这类明确的依赖清单，因此依赖管理仍以 README 中的手工安装说明为主。

### 3.2 核心 AI / CV 依赖

结合 `README.md` 和实际脚本，当前项目依赖主要包括：

- **Ultralytics YOLO**：训练、加载 `.pt` 权重、推理、验证、模型导出。
- **PyTorch**：YOLOv11 背后的训练/推理框架基础。
- **OpenCV (`cv2`)**：图像读取、尺寸处理、颜色空间转换。
- **NumPy**：张量/数组处理。
- **Matplotlib**：可视化推理框、统计展示、生成示意图。
- **Pandas**：验证结果汇总表输出。
- **PyYAML**：读取 `data.yaml`。
- **PIL**：部分图片处理流程中使用。

README 中列出的安装示例：

```bash
pip install ultralytics opencv-python matplotlib torch torchvision
pip install jupyter notebook albumentations numpy
```

### 3.3 当前输出模型格式

仓库中已经存在或被脚本支持的模型格式包括：

- **PyTorch `.pt`**：训练主格式，也是当前脚本默认使用的格式。
- **ONNX `.onnx`**：用于跨平台部署，是未来浏览器侧推理的重要中间格式。
- **CoreML `.mlpackage`**：用于苹果生态部署。

这是后续迁移到 **WASM + WebGPU** 方案时最关键的已有基础之一，因为浏览器端方案通常不会直接消费 `.pt`，而更可能基于 **ONNX** 或等价的导出结果。

## 4. 目录与模块职责

### 4.1 `trained_models_v2/`

当前可见内容：

- `yolo11n_best.pt`
- `yolo11s_best.pt`
- `yolo11m_best.pt`

这是当前仓库里最明确的“**已训练最佳权重**”集合。多个脚本默认直接从这里加载模型，例如：

- `inference_validation.py`
- `generate_inference_examples.py`
- `create_labeled_demo.py`
- `validate_models.py`

### 4.2 `models/`

按模型规模划分：

- `models/nano/`
- `models/small/`
- `models/medium/`
- `models/large/`

其中包含：

- `.pt` 训练权重
- `.onnx` 导出模型
- 部分目录下已有 `.mlpackage`

这说明 `models/` 更像“**多格式分发目录**”，而 `trained_models_v2/` 更像“**当前主线实验结果目录**”。

### 4.3 `scripts/`

当前包含两个格式转换脚本：

- `scripts/convert_yolo_to_onnx.py`
- `scripts/convert_yolo_to_coreml.py`

它们的职责很明确：

- 加载 `.pt` 模型
- 调用 `Ultralytics YOLO.export(...)`
- 导出为 ONNX 或 CoreML
- 支持单文件和批量转换

这部分是未来浏览器部署链路的关键桥梁，因为 **浏览器侧推理路线通常要从导出模型开始，而不是从 Python 训练权重直接开始**。

### 4.4 `notebooks/`

当前目录中可以看到：

- `yolo.ipynb`
- `yolo_predict.ipynb`
- `yolo_with_yolo_data.ipynb`
- `data_labeling/`
- `data_processing/`
- `data.yaml`
- `data/`

这表明仓库的数据处理与训练流程相当依赖 Notebook，职责大致包括：

- 数据标注与标签转换
- 数据增强
- 数据预处理
- YOLO 训练实验
- 预测与结果分析

其中有一个对后续改造影响很大的现象：

- `yolo_with_yolo_data.ipynb` 所代表的训练流，说明项目至少存在一条“**将 38 类中的 `UNKNOWN` 去掉并重映射为 37 类**”的训练路径。

这意味着后续做 Web 化时，不能只参考 README 或演示脚本里的硬编码类别表，必须重新确认“真正用于训练和导出的类别定义”。

### 4.5 根目录验证/演示脚本

#### `validate_models.py`

职责：

- 加载多个 `.pt` 模型
- 调用 `model.val(...)`
- 统计 `precision / recall / mAP50 / mAP50-95 / 模型大小`
- 生成 `model_validation_results.csv`

#### `generate_inference_examples.py`

职责：

- 加载多个训练好的 YOLOv11 模型
- 对测试图片进行推理
- 生成可视化结果图
- 生成不同模型之间的对比图
- 输出 `inference_summary.txt`

#### `inference_validation.py`

职责：

- 用指定模型对测试集图片做推理
- 读取对应 YOLO 标签作为 ground truth
- 计算 IoU、匹配预测框与标注框
- 统计 TP / FP / FN、precision、recall、分类正确率
- 输出图片级可视化与 `validation_statistics.txt`

#### `create_labeled_demo.py`

职责：

- 加载模型
- 在示例图上绘制检测框、英文牌名和置信度
- 生成对外展示更友好的演示图

#### `check_classes.py`

职责：

- 读取 `data.yaml`
- 遍历训练/验证/测试标签目录
- 检查类别 ID 是否越界、是否缺失
- 对比模型内嵌类别名和配置文件类别名是否一致

## 5. 当前推理与验证链路

从现有脚本来看，仓库的主推理链路可以概括为：

```text
训练好的 .pt 模型
    ↓
Ultralytics YOLO 加载
    ↓
对图片运行推理
    ↓
读取 boxes.xyxy / boxes.conf / boxes.cls
    ↓
脚本侧完成可视化、统计与结果输出
```

### 5.1 现有在线推理方式

现有脚本主要使用：

```python
from ultralytics import YOLO
model = YOLO(model_path)
results = model(image_path, verbose=False)
```

也就是说：

- **模型执行** 在 Ultralytics / PyTorch 体系中完成。
- **后处理结果读取** 由 Python 脚本完成。
- **可视化与统计** 也都在 Python 脚本里完成。

### 5.2 ONNX 推理示例

README 中还给出了 `onnxruntime` 的基本调用示例，说明项目已经具备“从 PyTorch 转向导出推理图”的意识，这对后续浏览器化非常重要。

## 7. 数据与类别配置现状

这是当前仓库里最值得单独说明的一部分，因为它直接影响后续工程改造。

### 7.1 类别定义存在多处不一致

当前可以观察到以下几套类别定义：

1. `README.md` 中写的是 **38 类**，并包含 `UNKNOWN`。
2. `inference_validation.py` / `generate_inference_examples.py` / `create_labeled_demo.py` 中硬编码的类别列表也是 **38 类**，包含 `UNKNOWN`。
3. `notebooks/data.yaml` 当前显示的是 **37 类**，只包含 `0m / 0p / 0s`，**没有 `UNKNOWN`**。
4. `notebooks/data/processed/data.yaml` 当前内容是：
   - `nc: 1`
   - `names: [class_0]`

这说明当前仓库至少存在以下问题之一：

- 配置文件未统一更新。
- 某些 YAML 文件是阶段性产物或占位文件。
- 训练、验证、展示三条链路对“真实类别定义”的依赖并不一致。

### 7.2 为什么这很重要

如果后续要改造成 **WASM + WebGPU** 的浏览器项目，那么以下内容都必须统一：

- 模型输出类别顺序
- 类别总数
- 前端显示标签
- 后处理中的类别映射
- 演示样例中的文字说明

否则会出现“模型推理成功，但前端显示标签错位”的问题。

## 8. 工程组织特点

### 8.1 当前优势

这个仓库已经具备几个非常有价值的基础：

- 有多个训练完成的模型权重。
- 已经有 ONNX 导出脚本和现成 ONNX 文件。
- 有较完整的验证脚本与结果输出。
- 有可视化示例图，适合后续前端 Demo 化。

### 8.2 当前短板

与“面向浏览器部署的工程”相比，当前仓库还有明显缺口：

- 没有前端工程结构。
- 没有 TypeScript / JavaScript 运行时层。
- 没有统一的依赖锁定文件。
- 没有面向 Web 的模型加载、推理、渲染与资源管理代码。
- 数据配置和类别定义存在不一致。
- 部分 YAML 中出现 Windows 绝对路径，不利于跨环境复现。
- README 中描述的部分目录与当前仓库实况并不完全一致，说明文档与代码已经出现漂移。
- 仓库中未提交完整训练/验证数据集，现有脚本依赖的 `notebooks/data/processed/train|val|test/...` 并不能仅靠当前 Git 内容完全复现。

## 9. 适合如何理解这个仓库

更准确地说，这个项目现在不是一个“完整的 Web 产品”，而是一个：

> **以麻将牌检测为目标的 YOLOv11 模型训练、验证、导出与演示仓库。**

如果后续要转为 **WASM + WebGPU** 项目，最合理的思路不是推翻现有内容，而是把当前仓库拆成两部分看待：

1. **模型工厂层**：继续保留 Python / Ultralytics / Notebook，用于训练、验证、导出。
2. **Web 推理层**：新增前端工程，负责浏览器中的模型加载、预处理、推理、后处理与可视化。

## 10. 对后续改造最有价值的现有资产

从迁移角度看，当前最值得保留和复用的资产是：

### 10.1 可直接复用

- `trained_models_v2/*.pt`：作为主模型来源。
- `models/**/*.onnx`：作为 Web 侧推理的候选模型资产与历史导出参考。
- `scripts/convert_yolo_to_onnx.py`：继续承担模型导出职责。
- `model_validation_results.csv`：作为模型选择依据。
- `inference_examples/`、`inference_validation/`：作为前端回归测试的样例材料。

### 10.2 需要整理后才能复用

- `data.yaml` 相关配置
- 类别名映射
- 推理阈值配置（如 `conf=0.3`、IoU 阈值等）
- Ground truth 与展示标签的统一格式
- `models/**/*.onnx` 与 `trained_models_v2/*.pt` 的对应关系

### 10.3 基本不能直接复用到浏览器运行时

- 基于 `Ultralytics YOLO(...)` 的 Python 推理调用
- Python 侧 Matplotlib 可视化逻辑
- 依赖本地文件系统路径的 Notebook 流程

此外，现有 `models/` 下的 ONNX 文件虽然很有价值，但更稳妥的做法不是直接把它们当作正式发布产物，而是：

- 先从 `trained_models_v2/` 中选定一个主模型；
- 再重新导出一份“面向浏览器发布”的标准 ONNX；
- 同时生成类别、阈值、输入尺寸、输出定义等配套元数据。

## 11. 总结

当前 Mahjong-YOLO 仓库的技术本质可以概括为：

- **核心语言**：Python
- **核心模型框架**：Ultralytics YOLOv11 / PyTorch
- **图像处理**：OpenCV + NumPy
- **可视化与分析**：Matplotlib + Pandas
- **实验形态**：Jupyter Notebook 驱动
- **部署过渡格式**：ONNX / CoreML

它已经具备“**从模型训练仓库向浏览器推理仓库演进**”所需的一部分关键资产，尤其是：

- 已训练权重
- 已导出 ONNX
- 已有验证样本和评估结果

但在真正进入 **WASM + WebGPU** 之前，必须先补齐以下工程基础：

- 统一类别与配置
- 明确 Web 运行时栈
- 重写推理前后处理链路
- 建立浏览器侧资源、线程与渲染组织方式

这也是下一份《WASM + WebGPU 改造计划》要解决的问题。
